## 今天修复

### 1. TypeScript / build 失败

6 月 8 日提交后，远端 CI 失败，本地 `typecheck` / `build:demo` 也失败。根因：

- `InMemorySoulStore.serialize()` 返回类型没有声明 `credentials`。
- `deserialize()` 给 `RuntimeSession` 显式传入 `undefined` 字段，违反 `exactOptionalPropertyTypes`。

修复：

- `serialize()` 返回类型补 `credentials`。
- `deserialize()` 只在字段存在时写入 `soulSnapshotId`、`nodeContext`、`dailyMessageCount`、`lastMessageDate`。

### 2. Credential 作用域风险

发现并修复：

- `deleteUserScopedData()` 中 `if (userId === userId)` 恒真，可能删除所有 credentials。
- `storeCredential()` 现在会先 `requireUser(userId)`，防止凭据绑定不存在的 user。
- 注册流程不再生成独立 `generateUserId()`；改为使用 `store.createUser(email)` 返回的真实 `user.id` 写 credential 和 JWT。
- 注册成功后会 `persistIfEnabled()`。

新增测试：

- `soul-scope.test.ts`：删除用户 A 时只删除 A credential，不影响 B credential。
- `persistence.test.ts`：credentials 保存/加载后 userId 不变化。

## 仍需注意

### 登录还没有真正接入用户自己的 Soul

当前注册/登录只解决了 credential 和 JWT；聊天仍使用 A/B demo fixture。下一步不能直接宣称“真实用户系统完成”。

下一步应该做：

1. 登录后根据 auth userId 创建/读取该用户自己的 Persona。
2. 用户端 chat 只使用自己的 `userId + personaId`。
3. A/B demo 保留在开发者验证页，不能和真实用户入口混在一起。
4. 对所有 `/api/chat`、Memory、Node、Proposal mutation 增加 auth-aware scope 路径。

### Render 免费版文件系统不可依赖

`NNZ_DB_PATH=./nnz.db` 适合本地 demo。Render 免费 Web Service 的文件系统不适合作为生产持久化。生产应转 Render PostgreSQL 或外部 Postgres。

### 下一步优先级

1. 进入 auth user → private Soul 的真实数据流，不要继续扩官网文案或微信接入。
2. 登录后根据 auth userId 创建/读取该用户自己的 Persona。
3. `/api/chat`、Memory、Node、Proposal mutation 全部走 auth-aware `userId + personaId`。
4. A/B demo 保留为开发者验证页；真实用户入口不能暴露 `SoulVersion`、`Proposal`、scope、A/B fixture 等内部机制。
