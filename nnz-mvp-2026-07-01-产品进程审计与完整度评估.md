## Step 2.19 审核结论

Step 2.19 的目标是合理的：把 disposable DB 验收从“手动跑两个 integration test + 手动试 protected execution”收敛成：

```bash
npm run migration:smoke -- --database-url-env NNZ_POSTGRES_INTEGRATION_URL --confirm RUN_POSTGRES_SCOPED_MIGRATION_SMOKE
```

已覆盖：

- 构造双 user / 双 persona fixture。
- 执行 scoped migration 两次，验证 executor 幂等。
- 通过 `PostgresScopedSoulRepository` 读回 session、snapshot、memory、conversation、proposal、credential。
- 验证 cross-scope node conversation 被拒绝。
- 验证 OpsAudit row 写入。
- 删除 user A 后验证 scoped rows cascade delete。
- 验证 sibling scope preserved。
- finally 尝试清理 fixture users / audit rows。
- stdout / failure output 不输出 DB URL、memory/chat、credential hash、row payload 或 raw DB error details。

已记录验证：

## Step 2.19 风险点

### P1：仍可能误连生产库别名

`migration:smoke` 限制的是 env 名称必须为 `NNZ_POSTGRES_INTEGRATION_URL`，但没有检查这个 env 的值是否和 `DATABASE_URL` / `NNZ_POSTGRES_URL` 相同。

如果有人把 `NNZ_POSTGRES_INTEGRATION_URL` 错填为 Render production URL，命令仍会连接、建表、写 fixture、删除 fixture。虽然 fixture 使用独立 user id 且会 cleanup，但这仍违背“disposable DB only”的安全意图。

建议补强：

- 如果 `env.NNZ_POSTGRES_INTEGRATION_URL === env.DATABASE_URL`，拒绝执行。
- 如果 `env.NNZ_POSTGRES_INTEGRATION_URL === env.NNZ_POSTGRES_URL`，拒绝执行。
- 错误信息不打印任何 URL 值。

同类 guardrail 也应补到 `migration:execute`。

### P2：`pool.end()` 失败可能绕过 sanitized error

`runMigrationSmokeCommand()` 的 finally 中直接 `await pool.end()`。如果 close 阶段抛错，可能绕过原本的 sanitized failure output。

建议补强：

- 把 `pool.end()` 包进 catch。
- close failure 只输出固定的 sanitized close error，不输出 raw database details。

### P3：路线文档曾有轻微滞后

`nnz-mvp-2026-07-01-Step2-MigrationReadinessRoadmap.md` 和 `nnz-mvp/CLAUDE_CODE_HANDOFF.md` 的最新提交文本已同步为 `dff6a47 feat: add disposable migration smoke CLI`。后续如发现其他历史段落仍有旧 commit，只按 `git log -1 --oneline` 校准即可。

## 产品目标对齐

产品宪法重点是：

- 帮助用户离开，不是留下。
- AI persona time-frozen，不随用户成长。
- Covenant 节奏必须硬约束。
- 数据主权：用户可导出、可永久删除。
- 不做情感操控。
- 用户端不能暴露机制词和后台对象。

当前工程在“底层边界”上高度对齐：

- `userId + personaId` scope 被反复加固。
- Covenant ACTIVE / SEALED / NODE / GRADUATED 已有领域模型与测试。
- Memory 分层、Proposal review、Ops audit、RBAC、migration readiness 都在服务安全和数据主权。
- 机制泄露测试和 prompt/safety guard 已有基础。

但当前产品体验层还没有充分体现“帮助离开”：

- Graduation 作为产品成功指标，仍主要是 domain state，不是完整用户旅程。
- Export / delete 的用户自助闭环还不完整。
- Sealing / node restart 的用户体验尚未形成稳定 H5 / WeChat flow。
- 用户端还没有足够的告知、同意、边界说明和退出引导。

## 是否偏移或过度开发

存在“工程重心过度偏 migration readiness”的风险。

Step 2.9 到 Step 2.19 做了很多 migration 周边能力：planner、dry-run、row builder、executor、transaction、export、summary、execute CLI、readiness CLI、smoke CLI。由于这是 grief-tech 且最核心风险是跨用户 Soul 污染，前半段是合理的。但继续增加 migration 工具层，而不跑真实 snapshot / disposable DB，就会变成工具堆叠。

建议设停止线：

- 不再新增 migration CLI。
- 先修 P1/P2 guardrail。
- 然后用真实本地 snapshot 跑 `migration:readiness`。
- 再用 disposable DB 跑 `migration:smoke`。
- 没有上述真实验证结果前，不切线上 runtime persistence。

## 欠缺功能点

### 用户产品层

- WeChat OAuth / service account 真实接入未完成。
- 用户 onboarding / consent / memorial setup 仍是 demo 形态。
- Memory collection 还没有面向普通用户的低摩擦体验。
- Seal / Node / Graduation 缺完整前台流程。
- Export / permanent delete 缺用户自助体验和验收记录。
- Node 付费、订阅、lifetime buyout、B2B 都未进入 MVP 验证。
- Voice cloning 仍是后续能力，尚未进入合规/授权/撤销设计。

### 安全与合规

- 数据加密目前主要依赖平台/数据库；产品要求的 AES-256-GCM 应用层加密尚未落地。
- raw snapshot / readiness output 可能包含 memory、chat、credential hash；`.gitignore` 目前没有显式忽略 `*snapshot*.json`、`*migration*report*.json` 等本地导出物。
- `NNZ_OPS_VIEWER_TOKEN` / `NNZ_OPS_OPERATOR_TOKEN` / `NNZ_OPS_ADMIN_TOKEN` 云端 smoke 仍未验证。
- Crisis / high-risk grief 场景只有基础 guard，尚缺人工升级、热线地域化、审计和误伤处理流程。
- LLM prompt injection 目前有基础测试，但未形成端到端红队用例。
- 数据删除的法律语义、备份保留、审计保留边界尚未形成 runbook。

### 运行与发布

- Render 当前仍是 snapshot JSONB persistence，尚未切 scoped tables runtime。
- disposable DB integration 尚未真实跑过。
- migration runbook 还缺“实跑前检查、失败回滚、验收证据、禁止项”。
- 当前 `/ops` 是重要后台，但还没有完整 production RBAC / session / IP allowlist / audit retention 策略。
