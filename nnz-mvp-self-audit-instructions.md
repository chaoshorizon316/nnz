## 审核清单

### 1. 通道对齐

原始产品方案的核心交互通道是**微信原生对话**（企业微信/服务号），展示官网是辅助。
- 当前代码库中微信接入的实现程度是多少？
- 如果确认走微信，当前 H5 前端代码（`public/index.html` 2,681 行 + `demo-server.ts`）有多少可以复用？
- 如果实际上走的是 H5/Web App 路线，产品方案文档是否已更新以反映这一决策？

### 2. 过度建设 vs 欠建设

- 列举所有 `src/tools/` 下的 CLI 命令名称和用途。
- 对于一个尚未有真实用户的 grief-tech MVP，哪些 CLI 工具是当前阶段必需的？哪些可以推迟？
- 对比你投入在"发布工程 / CLI / Ops 基础设施"的代码量和你投入在"用户能直接看到和使用的功能"上的代码量。差距是否合理？

### 4. 安全合规缺口

- 产品方案明确要求 AES-256 应用层加密——当前实现了吗？
- 用户协议、隐私政策、伦理承诺是否已起草？
- 未成年人保护机制是否已实现？
- 危机升级（人工审核、热线地域化、误伤处理）是否只停留在基础 guard？
- ICP 备案、PIPL 合规是否有任何进展？

### 5. 基础设施风险

- Render Free Postgres 已于 2026-07-11 过期。当前数据是否已备份？是否还有可用的数据库？
- 如果明天需要把产品部署到腾讯云（中国大陆），需要做多少工作？
- 当前是否有任何形式的自动化备份或灾难恢复方案？

### 6. 产品-用户验证

- 是否有任何真实丧亲用户测试过当前 H5 体验？
- "帮助用户离开"这个核心伦理目标是否有任何实证支撑？
- 封存节奏、节点定价、毕业时机——这些设计决策是否有用户研究基础，还是纯假设？

### 7. 工程架构自查

对照 `.agents/skills/system-architecture/SKILL.md` 中的架构宪法，逐条检查：
- `userId + personaId` 作用域是否在所有访问路径上强制执行？

### 8. 工程“体温”检查

- 预估：如果一个新的资深工程师现在接手这个代码库，需要多少小时才能独立做一个小的用户端改动（比如加一个 H5 按钮）？
- `CLAUDE_CODE_HANDOFF.md` 77KB，`nnz-mvp-CURRENT-STATE.md` 59KB——这些交接文档本身是否已经成了认知负担？
- 35 个测试文件、14 个 CLI 工具、多层抽象（adapter → repository → executor → smoke CLI → validation suite）——这个架构复杂度对于一个"帮丧亲者在微信里和 AI 聊天"的产品是否过度？
